# Modèle C4 — diagrammes

Cette page décrit l'architecture du projet **Michelin Vélo** avec le
[modèle C4](https://c4model.com/) : on zoome progressivement du contexte général
(niveau 1) jusqu'aux composants internes (niveau 3), puis on projette le tout sur
l'infrastructure réelle ([Schémas d'infrastructure](infrastructure.md)).

Les diagrammes ci-dessous sont rendus à la volée (Mermaid). La **source de vérité
C4** est par ailleurs versionnée en [Structurizr DSL](#source-structurizr) :
`diagrams/structurizr/workspace.dsl` dans ce repo.

## Niveau 1 — Contexte système

Qui utilise la boutique Michelin Vélo et avec quels systèmes externes elle dialogue.

```mermaid
C4Context
  title Niveau 1 — Contexte système · Michelin Vélo

  Person(client, "Client", "Achète des produits de la gamme Michelin Vélo")
  Person(admin, "Administrateur", "Gère le catalogue, les commandes et le contenu")
  Person(dev, "Développeur / DevOps", "Livre le code et opère la plateforme")

  System(shop, "Plateforme e-commerce Michelin Vélo", "Boutique en ligne premium : storefront, back-office et API, livrés en continu sur AKS")

  System_Ext(cloudflare, "Cloudflare DNS", "Résolution DNS de *.michelin.shost.fr (DNS-only)")
  System_Ext(letsencrypt, "Let's Encrypt", "Autorité de certification (ACME / TLS)")
  System_Ext(github, "GitHub + GHCR", "Code source, CI/CD et registre d'images privé")
  System_Ext(azure, "Microsoft Azure", "AKS, Key Vault, identité managée")

  Rel(client, shop, "Navigue et achète", "HTTPS")
  Rel(admin, shop, "Administre", "HTTPS")
  Rel(dev, github, "Push, Pull Requests", "git")
  Rel(dev, shop, "Opère (Argo CD)", "HTTPS")

  Rel(shop, cloudflare, "Est résolu via", "DNS")
  Rel(shop, letsencrypt, "Obtient ses certificats", "ACME DNS-01")
  Rel(shop, github, "Tire images & manifs GitOps", "HTTPS")
  Rel(shop, azure, "S'exécute sur / lit ses secrets", "API Azure")

  UpdateLayoutConfig($c4ShapeInRow="2", $c4BoundaryInRow="1")
```

| Conteneur | Techno | Rôle |
| --- | --- | --- |
| **Storefront** | Next.js (standalone, port 3000) | Boutique client |
| **Back-office** | Frontend (port 80) | Administration |
| **API** | Backend (port 8080) | Données, commandes, paiement |
| **APISIX** | Apache APISIX (Helm) | Gateway unique, TLS, routage par hôte |
| **Argo CD** | argo-cd (Helm) | Réconciliation GitOps |
| **Argo Rollouts** | argo-rollouts (Helm) | Canari 25 % → 50 % → 100 % |
| **cert-manager** | jetstack (Helm) | Certificat wildcard Let's Encrypt (DNS-01) |
| **External Secrets** | ESO (Helm) | Sync. Azure Key Vault → Secrets K8s |

## Niveau 3 — Composants : la chaîne de livraison

Zoom sur le **moteur GitOps** (Argo CD) et la façon dont une application arrive
dans le cluster, du commit à la mise en production.

```mermaid
flowchart TB
  dev(["👤 Développeur"]):::person

  subgraph ci["GitHub Actions (CI réutilisable)"]
    quality["quality<br/><i>lint · types · tests · audit</i>"]:::comp
    security["security<br/><i>Trivy fs (vulns + secrets)</i>"]:::comp
    build["build<br/><i>buildx · SBOM · provenance<br/>Trivy image · cosign · push GHCR :sha</i>"]:::comp
    release["release<br/><i>semantic-release · retag :version</i>"]:::comp
    bump["bump-prod<br/><i>PR auto sur gitops/envs/prod</i>"]:::comp
  end

  subgraph argocd["Argo CD (namespace argocd)"]
    root["platform-root<br/><i>app-of-apps</i>"]:::comp
    appsetProd["ApplicationSet prod<br/><i>liste : storefront/back-office/api</i>"]:::comp
    appsetPrev["ApplicationSet previews<br/><i>générateur Pull Request</i>"]:::comp
    appController["Application Controller<br/><i>sync + selfHeal</i>"]:::comp
  end

  ghcr["GHCR :sha / :version"]:::ext
  gitops["hackation-ops<br/>gitops/envs · appsets · chart"]:::ext
  cluster["Namespaces prod / pr-&lt;n&gt;<br/><i>Rollout · Service · ApisixRoute</i>"]:::container

  dev -->|push / PR| quality
  quality --> security --> build --> release --> bump
  build -->|"image :sha"| ghcr
  bump -->|"commit image.tag"| gitops

  root -->|"déclare"| appsetProd
  root -->|"déclare"| appsetPrev
  gitops -->|"valeurs + chart Helm"| appController
  appsetProd --> appController
  appsetPrev -->|"PR labellisée preview"| appController
  appController -->|"applique manifs"| cluster
  cluster -. "pull image" .-> ghcr

  classDef person fill:#08427b,stroke:#052e56,color:#fff;
  classDef comp fill:#85bbf0,stroke:#5d82a8,color:#000;
  classDef container fill:#438dd5,stroke:#2e6295,color:#fff;
  classDef ext fill:#999999,stroke:#6b6b6b,color:#fff;
```

**Lecture :** la CI de l'app produit une image signée (`:sha` puis `:version`),
puis ouvre une PR de *bump* sur `hackation-ops`. Argo CD détecte le commit et
réconcilie le cluster — c'est le seul chemin vers la prod. Détails du flux :
[GitOps & flux de déploiement](gitops.md).

## Source Structurizr

Le modèle C4 canonique est décrit en **Structurizr DSL** dans ce repo :

```
diagrams/structurizr/workspace.dsl
```

Pour le rendre en local (vues interactives, export PNG/SVG/PlantUML) :

```bash
docker run -it --rm -p 8080:8080 \
  -v "$(pwd)/diagrams/structurizr:/usr/local/structurizr" \
  structurizr/lite
# puis ouvrir http://localhost:8080
```

Le DSL contient les vues *SystemContext*, *Container*, *Component* et un
*Deployment* (projection sur Azure / AKS). Mettez-le à jour en même temps que les
diagrammes Mermaid de cette page pour qu'ils restent cohérents.
